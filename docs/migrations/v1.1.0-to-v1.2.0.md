# Migrating from v1.1.0 to v1.2.0

v1.2.0 preserves the public local adoption entry point while strengthening the formal downstream contract. The important migration rule is simple: regenerate adoption artifacts with v1.2.0 rather than editing or reusing v1.1.0 Preview files and receipts.

## Before upgrading

- Keep the existing v1.1.0 Preview and receipts only if they are needed for historical audit.
- Record the target repository branch and commit that the new Preview must bind.
- Prepare and review a formal scope file in JSON or YAML. Do not infer installation or activation permission from that scope.

## Migration procedure

1. Upgrade or check out AI Agent Project Governance v1.2.0.
2. Confirm that the framework checkout is a clean Git checkout. Modified or staged generator inputs fail closed.
3. Generate a fresh read-only plan with the formal scope:

   ```bash
   python3 scripts/agent_adopt.py \
     --project-root /path/to/project \
     --format json \
     --scope-file /path/to/reviewed-scope.yaml \
     dry-run
   ```

4. Generate a new external Preview and review the canonical scope, writeset, pre-install hashes, rollback manifest, Runtime bytes, and provenance receipt together.
5. Do not hand-edit the generated plan, manifests, candidate, or receipt. Regenerate the Preview if the target branch, target commit, formal scope, framework commit, or generator inputs change.
6. Repeat the explicit confirmation and installation-approval workflow over the unchanged v1.2.0 artifact set.
7. Request activation separately after an approved installation. Preview generation and installation do not grant activation authority.

## Why old receipts are rejected

v1.1.0 receipts do not carry the Git `HEAD` blob basis, provenance contract version, and related exact-input bindings required by provenance v2. There is no safe in-place conversion because adding those fields after generation would not prove the original generator inputs. The supported migration is to regenerate the complete Preview with v1.2.0.

## What remains compatible

- Read-only `dry-run` remains available.
- Existing public CLI commands are retained.
- The framework remains local-only by default and does not add automatic network, Git write, deployment, installation, or activation authority.
- Old receipts remain valid as historical records, but not as input to the v1.2.0 formal downstream flow.

## Cross-platform verification

Git-blob provenance is designed to produce the same generator identity across LF and CRLF checkouts. For Windows/WSL verification, use a native Linux clone or a verified Git bundle. WSL may not be able to resolve the absolute `gitdir` stored by a Windows linked worktree.

## Rollback

If the v1.2.0 Preview has not been installed, discard the newly generated external Preview and remain on the prior framework version. If installation has begun, use the generated manual rollback assessment and follow the explicit recovery process; automatic deletion is not enabled.
