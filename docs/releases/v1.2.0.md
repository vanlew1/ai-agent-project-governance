# v1.2.0 — Deterministic Cross-Platform Adoption Governance

AI Agent Project Governance v1.2.0 strengthens the existing-project adoption path with a formal project-defined scope, deterministic pre-write artifacts, Git-blob provenance, and a bounded release gate. The release remains local-first: it does not add network execution, deployment, or automatic project authorization.

## Highlights

- Formal adoption scopes can be supplied through `--scope-file` and are enforced consistently during planning, confirmation, export, Runtime compilation, approval, and installation.
- One canonical writeset drives `INSTALL_WRITESET.json`, `PRE_INSTALL_HASHES.json`, and `ROLLBACK_MANIFEST.json` before target writes.
- Toolchain Provenance Binding v2 derives generator identity from clean Git `HEAD` blobs, fails closed for modified or staged generator inputs, and is deterministic across LF and CRLF checkouts.
- Formal artifacts use byte-stable UTF-8/LF output, exclusive creation, and atomic replacement where applicable.
- The canonical Release Gate uses an auditable 600-second default hang guard that can be configured from 60 through 3600 seconds.
- Immutable Git blob digests are cached and Git validation/blob reads are batched without relaxing provenance checks.

## Safety changes

- Pre-v1.2.0 adoption artifacts do not contain the provenance v2 contract and are rejected by the formal downstream flow.
- Generated confirmation and approval candidates remain authorization-neutral.
- A Preview is not an installation. Installation is not activation. Both still require separate, explicit Owner decisions over unchanged artifacts.
- The provenance binding detects ordinary artifact or generator-input changes; it is not a keyed anti-forgery signature.

## Upgrade summary

1. Upgrade the framework to v1.2.0.
2. Re-run `dry-run` with the reviewed formal scope file.
3. Generate a new adoption Preview; do not edit a generated plan or reuse an old receipt.
4. Repeat the confirmation and approval workflow over the new exact bytes.
5. Retain old receipts only as historical audit evidence.

See [Migrating from v1.1.0 to v1.2.0](../migrations/v1.1.0-to-v1.2.0.md) for the complete procedure.

## Compatibility

- Existing read-only `dry-run` usage remains available.
- No public CLI command was removed.
- The formal downstream flow requires freshly generated provenance v2 artifacts; old adoption receipts are intentionally not forward-compatible.
- Schema contract identifiers remain on their existing `1.0` lines while the required provenance-bound fields are enforced by the v1.2.0 toolchain.

## Known limitation

A Windows linked worktree can store an absolute Windows `gitdir` reference that WSL Git cannot resolve directly. Perform cross-platform revalidation from a native Linux clone or a verified Git bundle instead of opening that Windows linked worktree from WSL.

## Scope

This release upgrades the governance framework only. It does not install or activate the framework in another project and does not authorize downstream project work.
