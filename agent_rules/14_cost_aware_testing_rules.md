# 14 Cost Aware Testing Rules

- 高成本验证优先采用分层验证策略。
- 先做 smoke test、局部样本和 dry-run，再决定是否扩大范围。
- 当任务影响面高时，不能因为成本原因跳过必要验证。
- 在报告中明确说明采用了哪种成本控制策略。

## Token-Efficient Execution Contract

### 1. 总原则

在不降低正确性、数据安全、可回滚性和验收质量的前提下，Agent 必须优先减少：

- 重复读取、重复规则复述和无关上下文；
- 重复测试、过早备份和冗长中间报告；
- 阻塞状态下的无效施工；
- 与主线、其他分支或已有产物等价的重复建设。

节省 token 不得以删除安全门、降低测试质量或跳过验收为代价。

### 2. Gate 0：人工前置条件

凡任务涉及以下任一事项，必须先检查人工前置条件，再读取大量文件、修改代码、运行完整测试或备份数据库：

- 真实外部 API、connectivity probe、联网抓取或登录态；
- 正式数据库写入、production/evidence/pool 写入或迁移；
- 网络环境、VPN/TUN 状态或云服务连通性；
- 用户明确授权、不可逆操作或高成本操作。

如当前上下文中没有明确且仍有效的用户确认：

1. 立即停止施工。
2. 只请求缺失确认。
3. 不读取完整规则树、不修改代码、不运行完整测试、不备份正式数据库、不调用真实 API。
4. 不生成正式收口报告，只允许生成最小 checkpoint。

最小 checkpoint 仅包含：

- `task_name`
- `status = WAITING_FOR_USER_CONFIRMATION`
- `missing_confirmations`
- `live_requests = 0`
- `files_modified = 0`

不得先施工较长时间后再询问 Gate 0 条件。

### 3. 人工确认复用规则

已有用户确认只有在以下条件同时满足时才允许复用：

- 属于同一任务或明确续接任务；
- 环境没有被声明发生变化；
- 当前工作区和执行进程已重新核验；
- 确认内容覆盖本次所需全部条件；
- 没有新增高风险条件。

若无法确认其仍有效，必须重新询问，但只询问缺失项。不得重复询问用户已经在当前任务中明确确认且仍然有效的条件。

### 4. 最小读取原则

Agent 默认只读取：

1. 当前任务报告或交接文件；
2. 当前任务直接涉及的代码；
3. 当前任务直接涉及的测试；
4. 必要的任务卡、数据安全规则和项目特定规则。

默认禁止展开读取完整规则树、全仓库扫描、大量历史报告、重复读取当前上下文已给出的内容，或为证明合规而复述全部已读文件。发现冲突、缺失或异常时，才允许增量读取其他文件。

### 5. 已有资产复用门

施工前应检查当前主线报告、独立分支报告、manifest、审计工具、migration、测试、proof、backup、cache/version 工具等已有资产。

如已有可复用资产，优先复用并只核验路径、版本、哈希和适用性；不得重新开发或重新生成等价产物。

如资产位于其他分支或 worktree，不默认视为已同步；先记录来源 branch、commit、path、hash，再选择性同步必要文件，禁止盲目合并整个分支。

### 6. 测试分层规则

测试按以下顺序执行：

1. Level 1：新增或修改点定向测试。
2. Level 2：Level 1 通过后，统一运行一次相关模块回归。
3. Level 3：仅在架构级修改、跨模块修改、production 前验收、用户明确要求或最终总审计时运行全量回归。

修复失败测试后，先重跑失败集合；通过后再统一运行一次相关回归。禁止在每次小修后重复运行相同的完整测试集合，也不得连续重复运行已通过的同一组测试。

### 7. 数据库备份时机

正式数据库备份必须紧邻实际写入、迁移或真实执行。以下情况禁止过早备份：

- Gate 0 尚未通过；
- 任务仍可能因用户确认缺失而停止；
- 当前阶段只是只读审计；
- 代码和测试尚未通过；
- 当前阶段不会写正式数据库。

备份前必须确认下一步确实将执行正式写入。

### 8. 阻塞任务收口

如任务在真实执行前被阻塞，不得生成完整正式施工报告，只生成：

- 简短 checkpoint JSON；
- 阻塞原因；
- 已完成的必要准备；
- 下一步续接条件。

只有已执行真实操作、已修改正式数据、已完成有独立价值的代码施工、用户明确要求阶段性正式报告，或当前任务被明确冻结/终止时，才生成正式报告。

### 9. 输出压缩规则

Agent 中间回复只说明当前完成了什么、发现了什么阻塞、下一步做什么。

禁止重复输出完整规则列表、完整允许/禁止范围、历史阶段背景、用户已经给出的任务内容，或多次相同任务分类声明。任务边界只声明一次。

上下文压缩恢复后，只写：“继续沿用既定任务边界和已确认安全条件。” 不得完整重复前文。

### 10. Windows 执行环境规则

在已知 Windows 工作区中：

- 优先使用短命令；
- 避免超长 PowerShell 单命令；
- 已知 `apply_patch` 被 ACL 拒绝时，不重复尝试；
- 使用已验证的短 Python 编辑脚本；
- 大修改拆成少量明确步骤；
- 不因工具失败重复读取全部上下文。

### 11. 报告压缩规则

正式报告只保留：

1. 实际修改；
2. 实际执行；
3. 测试结果；
4. 数据状态变化；
5. 安全结果；
6. 异常；
7. 验收结论；
8. 下一步。

禁止重复历史架构说明、完整规则文件清单、已在前序报告中确认的背景、未发生的操作或大量无变化字段。

### 12. 施工指令默认格式

后续执行方案默认只包含：

1. 任务名称；
2. 推荐模型/智能强度；
3. 目标；
4. Gate 0；
5. 最小读取范围；
6. 允许修改范围；
7. 硬边界；
8. 执行步骤；
9. 必要测试；
10. 验收标准；
11. 报告路径。

架构设计、P0 修复、跨系统重构和最终总审计可按需要扩展。

### 13. 验收要求

新增或修改治理规则后必须确认：

- 真实 API 任务会先触发 Gate 0；
- Gate 0 缺失时不会修改代码或备份数据库；
- 已确认且仍有效的条件不会被重复询问；
- 小任务不会读取完整规则树；
- 测试按 Level 1 -> Level 2 -> Level 3 逐步升级；
- 阻塞任务只生成 checkpoint；
- 其他分支资产会先检查再施工；
- 新规则不会削弱数据安全与正式写入门禁。
